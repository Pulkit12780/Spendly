✨ Core Mandate: Spec-Driven Testing

Tests are written based on what the feature is supposed to do — its specification and user stories — never by reading the implementation code and mirroring it.

Observable Outcomes Only Test inputs, outputs, HTTP responses, redirects, session state, DB side-effects — all from the outside in.
No Logic Mirroring If the route does X, don't test that X is called. Test that the observable outcome matches the spec.
Ask First If the feature spec is ambiguous, ask clarifying questions before writing a single test.
▶

Test Writing Process

1
Identify the Feature Spec

Extract what the feature must do from the task description, route table, or user story.

2
Determine Test Categories

Map out happy path, validation failures, auth/authz, DB side effects, HTTP semantics, and edge cases.

3
Write the Tests

Follow the pytest standards below: fixtures, naming conventions, and assertion patterns.

4
Self-Review Checklist

Verify naming, isolation, no hardcoded URLs, observable-only assertions before finalizing.

Pytest Standards

File: tests/test_<feature>.py

import pytest
from app import app
from database.db import init_db

@pytest.fixture
def client():
    app.config['TESTING'] = True
    app.config['DATABASE'] = ':memory:'
    app.config['SECRET_KEY'] = 'test-secret'
    with app.test_client() as client:
        with app.app_context():
            init_db()
        yield client

Naming Convention

test_login_valid_credentials_redirects test_login_wrong_password_shows_error test_add_expense_missing_amount_400 test_logout_clears_session
🔍

Required Assertions

✓ HTTP status code — assert response.status_code == 200
✓ Redirect target — response.headers['Location']
✓ Response body — assert b'text' in response.data
✓ DB state — query test DB directly to verify rows changed
✓ Session state — client.session_transaction()
✓ Isolation — fresh DB per test, no state leakage
✓ No hardcoded URLs — use route path strings matching app.py

⚠

Never Do

✗ Read app.py to understand route logic and mirror it in tests
✗ Write tests for stub routes unless confirmed implemented
✗ Hardcode database file paths — use :memory: or tmp_path
✗ Use requests or real HTTP — always use Flask's test_client()
✗ Install new packages — only what's in requirements.txt
✗ Write vacuous tests — no assert True
✗ Skip negative/edge case tests — they matter as much as happy path
